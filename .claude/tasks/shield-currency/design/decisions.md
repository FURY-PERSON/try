# Решения: Валюта "Щит"

## 1. Хранение баланса — поле в User vs отдельная таблица

**Решение:** Поле `shields: Int` в модели User.

**Обоснование:**
- Простота — нет отдельной таблицы транзакций
- Атомарность через `UPDATE WHERE shields > 0`
- Нет потребности в истории транзакций на данном этапе
- Совпадает с паттерном `totalScore`, `currentStreak` — всё в User

**Риск:** Невозможность аудита транзакций. **Митигация:** Можно добавить таблицу ShieldTransaction позже, если потребуется.

## 2. Shield validation — клиент vs сервер

**Решение:** Серверная валидация обязательна (требование задачи №8).

**Реализация:**
- Клиент передаёт `useShield: true` в `answerQuestion`
- Сервер атомарно проверяет `shields > 0` и списывает
- Если shields === 0, щит не применяется (streak сбрасывается)
- Клиент получает `shieldUsed: true/false` в ответе

**Подход:** Оптимистичный UI — клиент показывает анимацию сразу, сервер подтверждает. Если сервер отклонит — streak на клиенте корректируется.

## 3. Интеграция в answerQuestion vs отдельный endpoint

**Решение:** Расширить существующий `POST /v1/questions/:id/answer` полем `useShield`.

**Обоснование:**
- Атомарность — щит и ответ обрабатываются в одной транзакции
- Нет race condition между "использовать щит" и "обработать ответ"
- Отдельный endpoint `/shields/use` оставить для экстренного списания (необязательно для MVP)

## 4. Начисление за daily set — в submitDailySet vs отдельно

**Решение:** В `submitDailySet` — начислять +3 щита если ≥50% правильных.

**Обоснование:** Данные о проценте правильных уже есть в момент submit. Добавляется одна проверка.

## 5. Начисление при первом заходе (5 шитов)

**Решение:** При `register()` — устанавливать `shields: 5`. Для существующих — миграция `UPDATE "User" SET shields = 5 WHERE shields = 0`.

## 6. Shield state на клиенте

**Решение:** Новый Zustand store `useShieldStore` (без persist — баланс всегда с сервера).

**Обоснование:**
- Баланс синхронизируется из `home/feed` при каждом заходе
- `shieldActive` — локальный флаг текущей игровой сессии
- Guideline flags — в `useAppStore` (persist), как `hasSeenSwipeHint`

## 7. Rewarded video для щитов

**Решение:** Переиспользовать `useRewardedAd()` hook и паттерн из `DisableAdsModal`.

**Flow:**
1. Пользователь нажимает "Посмотреть видео" в модалке
2. `showForReward()` → Unity rewarded ad
3. После просмотра → `POST /v1/shields/reward`
4. Обновить баланс в store

## 8. Guideline (пункт 5 задачи)

**Решение:** Два новых boolean flags в `useAppStore`:
- `hasSeenStreakGuideline` — пояснение про стрик
- `hasSeenShieldGuideline` — пояснение про щит

**Триггер:** Первый раз когда streak сбросился в 0 и затем стал > 0.
**UI:** OverlayModal с двумя шагами.

## Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Race condition при списании | Низкая | Атомарный UPDATE WHERE shields > 0 |
| Обход клиентом | Средняя | Серверная валидация обязательна |
| Abuse rewarded video | Низкая | Rate limit на /shields/reward (1 раз в 60 сек) |
| Десинхрон баланса клиент-сервер | Средняя | Баланс обновляется из API при каждом ответе и из feed |
