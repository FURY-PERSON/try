# Решения: Факт дня

## 1. Где хранить факт дня

**Решение:** Поле `factOfDayQuestionId` в модели `DailySet` (nullable String).

**Обоснование:** Факт дня привязан к конкретному daily set. Не нужна отдельная таблица — это просто указатель на один из вопросов сета. Nullable позволяет работать без fact of day для старых/fallback сетов.

**Альтернатива отвергнута:** Поле `isFactOfDay` в `DailySetQuestion` — избыточно, нужен constraint "только один".

## 2. Авто-выбор vs только ручной

**Решение:** Гибрид. Админ может задать `factOfDayQuestionId`. Если не задан — при `submitDailySet` сервер автоматически выбирает вопрос с наименьшим `timesCorrect / timesShown` из вопросов этого сета.

**Обоснование:** Не требует обязательных действий от админа. Статистика `timesCorrect/timesShown` уже есть и отражает "контринтуитивность".

## 3. Когда показывать факт дня

**Решение:** В составе ответа `submitDailySet`. На мобилке — на экране результатов (ResultsModal), как дополнительная секция.

**Обоснование:** Не нужен отдельный экран/модал. Пользователь уже видит результаты — логично показать факт дня там же. Минимум навигационных изменений.

## 4. Расчёт wrongPercent

**Решение:** `wrongPercent = Math.round((1 - timesCorrect / timesShown) * 100)`. Берётся из поля Question на момент submit.

**Обоснование:** Данные уже обновляются атомарно при каждом ответе. Актуальны. Не нужен отдельный запрос.

## 5. Шеринг

**Решение:** Новая функция `shareFactOfDay()` в `mobile/src/utils/share.ts`. Формат текста из задачи.

## Риски и митигации

| Риск | Митигация |
|------|-----------|
| Question.timesShown = 0 (нет статистики) | Показывать wrongPercent = 0, или не показывать factOfDay |
| factOfDayQuestionId указывает на вопрос не из сета | Валидация в admin DTO + проверка при submit |
| Fallback daily set (без id) | factOfDay = null для fallback |
