# Feature Pipeline: research → design → plan → implementation

## Входные данные

Задача: $ARGUMENTS

## Инструкции

Ты — оркестратор полного пайплайна разработки фичи. Проведи задачу через все этапы последовательно, останавливаясь на ревью пользователя после каждого.

Прочитай правила перед началом:
- `.claude/rules/git-conventions.md`
- `.claude/rules/server-conventions.md` (если задача для server)
- `.claude/rules/web-conventions.md` (если задача для web)
- `.claude/rules/mobile-conventions.md` (если задача для mobile)

---

## Этап 0: КОНТЕКСТ

### 0.1 Определение области

Определи какие части проекта затрагивает задача. Если неясно — ОБЯЗАТЕЛЬНО спроси:

> Какие части проекта затрагивает эта задача?
> 1. **server** — NestJS API (server/)
> 2. **web** — Admin panel (web/)
> 3. **mobile** — Expo приложение (mobile/)
> 4. **full-stack** — несколько частей

---

## Этап 1: RESEARCH

### 1.1 Подготовка

1. Определи `feature-name` из задачи (короткое имя на английском, kebab-case)
2. Создай директорию `.claude/tasks/{feature-name}/research/`
3. Если задача — путь к файлу, прочитай файл

### 1.2 Валидация требований

Проведи интервью с пользователем — задавай вопросы по каждому неясному аспекту задачи. Если вопрос можно прояснить исследованием кодовой базы — исследуй код вместо того чтобы спрашивать.

### 1.3 Исследование кодовой базы

Определи области для исследования и запусти саб-агенты `researcher` (Explore, haiku) параллельно для каждой области.

### 1.4 Консолидация

Собери результаты в `.claude/tasks/{feature-name}/research/research_result.md`.

Формат:
```markdown
# Исследование: {название задачи}
## Область: {server | web | mobile | full-stack}
## Задача
{описание}
## Затрагиваемые модули
- `путь`: {описание}
## Существующие паттерны
- {паттерн}: `путь/файл.ts:строка`
## Зависимости и интеграции
## Ключевые файлы
## Модель данных (Prisma)
## API контракты
```

Содержит ТОЛЬКО факты. Никаких рекомендаций.

### 1.5 ПАУЗА: Ревью исследования

Покажи пользователю краткую сводку найденного и спроси:

> Исследование завершено. Результат в `.claude/tasks/{feature-name}/research/research_result.md`.
>
> **Область:** {area}
> **Найдено:** {краткая сводка — модули, файлы, паттерны}
>
> Проревьюй результат. Варианты:
> 1. **Продолжить** — перехожу к дизайну
> 2. **Доисследовать** — укажи что не хватает
> 3. **Стоп** — завершить на этом этапе

Жди ответа. НЕ продолжай без подтверждения.

## Этап 2: DESIGN

### 2.1 Создание дизайна

Создай 4 файла в `.claude/tasks/{feature-name}/design/`:

1. **architecture.md** — диаграммы компонентов (Mermaid), потоков данных, sequence-диаграммы
2. **contracts.md** — API-контракты, Prisma-модели, DTO
3. **decisions.md** — ключевые решения с обоснованием, риски и митигации
4. **test_strategy.md** — критерии приёмки, unit/integration/ручные тесты

### 2.2 Compliance check

Запусти агент `compliance-reviewer` для проверки дизайна. Если есть нарушения — исправь.

### 2.3 ПАУЗА: Ревью дизайна

> Дизайн завершён. Файлы в `.claude/tasks/{feature-name}/design/`.
>
> **Краткая сводка:**
> - Архитектура: {что затрагивается}
> - API: {какие контракты меняются}
> - Решения: {ключевые}
> - Тесты: {подход}
>
> Проревьюй. Варианты:
> 1. **Продолжить** — перехожу к планированию
> 2. **Правки** — опиши что изменить
> 3. **Стоп** — завершить на этом этапе

Жди ответа.

## Этап 3: PLAN

### 3.1 Составление плана

Создай `.claude/tasks/{feature-name}/plan/plan.md` с разбиением на фазы.

Критерии фазы:
1. **Атомарность** — код компилируется после фазы
2. **Одна область** — server / web / mobile
3. **Размер** — не более 5-7 файлов
4. **Независимость** — фазы без зависимостей могут идти параллельно
5. **Порядок** — Prisma → server → web/mobile

Формат каждой фазы:
```markdown
## Фаза N: {название}
- **Область:** server | web | mobile | prisma
- **Зависит от:** Фаза X (или "нет")
- **Файлы:** {список}
- **Что сделать:** {детально, с паттернами из research}
- **Критерии приёмки:** {чеклист}
- **Коммит:** `feat(scope): описание`
```

Включи граф зависимостей (Mermaid).

### 3.2 ПАУЗА: Ревью плана

> План завершён: **{N} фаз**. Файл: `.claude/tasks/{feature-name}/plan/plan.md`.
>
> **Фазы:**
> {список фаз с зависимостями}
>
> Проревьюй. Варианты:
> 1. **Продолжить** — перехожу к реализации
> 2. **Правки** — опиши что изменить
> 3. **Стоп** — завершить на этом этапе

Жди ответа.

## Этап 4: IMPLEMENTATION

### 4.1 Пофазная реализация

Для каждой фазы из плана (в порядке зависимостей):

**A. Разработка** — реализуй согласно плану и паттернам из research

**B. Code Quality ревью** — запусти агент `code-quality-reviewer`

**C. Compliance ревью** — запусти агент `compliance-reviewer`

**D. TypeScript проверка:**
```bash
cd {server|web|mobile} && npx tsc --noEmit
```

**E. Тестирование:**
```bash
cd server && npm run build && npm test   # для server
cd web && npm run build                   # для web
cd mobile && npm run lint                 # для mobile
```

**F. MCP-тестирование (ОБЯЗАТЕЛЬНО для UI-фаз):**
- **mobile** — использовать mobile-mcp
- **web** — использовать playwright-mcp

Если MCP недоступен — спросить пользователя. НЕ пропускать молча.

**G. Коммит фазы:**
- Формат: `<type>(<scope>): <описание на русском>`
- **Без Co-Authored-By**
- НЕ делать push

**H. Результат:**
- Все проверки пройдены + коммит создан → следующая фаза
- Проблемы → исправить → повторить (макс. 3 итерации)
- 3 неудачи → ОСТАНОВИТЬСЯ, сообщить пользователю

### 4.2 Финальная проверка

```bash
cd server && npm run build && npm test
cd web && npm run build
```

### 4.3 ПАУЗА: Финальный ревью

> Реализация завершена.
>
> **Результат:**
> - Фаз выполнено: {N}
> - Коммитов создано: {N}
> - Файлов изменено: {список}
> - Билд: PASSED/FAILED
> - Тесты: PASSED/FAILED
>
> Рекомендую запустить `/simplify` для пост-имплементационного ревью.
> Push нужно сделать самостоятельно.

## Обработка ошибок

| Сценарий | Действие |
|----------|----------|
| MCP для тестирования недоступен | Спросить: пропустить или подключить |
| Агент упал | Возобновить с последнего артефакта |
| Билд/линт fail | Исправить (макс. 3 попытки), затем эскалация |
| Git-конфликт | Остановиться, сообщить пользователю |
